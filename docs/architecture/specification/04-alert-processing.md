# 4. Feed processing and delivery

## Acquisition

For every configured feed, the watcher:

1. Parses the URL and requires HTTPS, the default port, an approved lowercase hostname, an empty user-info component, and a bounded path and query.
2. Resolves all A and AAAA records and rejects loopback, link-local, private, multicast, reserved, unspecified, or otherwise non-global addresses.
3. Connects to one validated address while preserving the approved hostname for TLS SNI and certificate verification.
4. Sends conditional `If-None-Match` and `If-Modified-Since` headers when available.
5. Rejects redirects, unsupported content types, oversized compressed or decoded responses, slow responses, and excessive items or item characters.
6. Parses XML with external entities, DTD processing, and network resolution disabled.
7. Stores a bounded raw response snapshot for replay and incident diagnosis.

DNS validation occurs for every connection. The HTTP client cannot silently re-resolve to an unchecked address.

One failed feed does not block successful feeds from the same run. Its validators remain unchanged, failure count rises, and feed-specific alarms apply.

## Normalization and coalescing

Accept configured `public_rss` and `public_atom` items with a resolvable canonical HTTPS item URL and nonempty title. Publication time is optional; `observed_at` is required and generated by the watcher. The normalized announcement uses `source_type = public_feed` because one announcement can merge provenance from feeds with different formats.

Normalization includes:

- Unicode normalization.
- HTML sanitization to plain text.
- Whitespace collapse.
- Bounded title and summary truncation with an explicit truncation marker.
- Canonical URL processing limited to scheme/host case, default port removal, fragment removal, and a reviewed list of tracking parameters.

Do not remove arbitrary query parameters or normalize paths in a way that can merge distinct resources.

Fetch all enabled feeds before emission. Group items by canonical URL, merge sorted provenance entries, and choose title, summary, and timestamps using a deterministic precedence rule recorded in code and fixtures. Conflicting content remains observable in metrics and snapshot evidence.

## Announcement revisions

`announcement_id` is SHA-256 of the canonical URL bytes. Content identity uses null-framed SHA-256 fields:

```text
content_fingerprint = SHA256(
  "announcement-content:v1\0" + normalized_title + "\0" + normalized_summary
)

revision_id = SHA256(
  "announcement-revision:v1\0" + announcement_id + "\0" + content_fingerprint
)
```

The implementation defines UTF-8 encoding, rejects null characters in framed values, and uses one framing helper for runtime and test vectors. A normalized title or summary change makes a revision. Publication timestamp corrections and new provenance alone update announcement state without delivery. `is_update` is true when announcement state already contains an earlier content revision for the same `announcement_id`.

## Matching

Matching evaluates title and summary fields selected by each risk rule.

1. Find service aliases using normalized phrase boundaries.
2. Find positive and excluded risk terms using the same boundary rules.
3. Evaluate `any`, `all`, and `none`.
4. Require at least one service evidence span and one positive risk evidence span that do not overlap.
5. Produce one service/risk result with normalized lexical ordering for aliases and terms, configured field ordering, and the rule ID.

When two aliases refer to different services in the same item, evaluate each service independently. When several rules share a risk type, configuration validation rejects the ambiguity. Rule precedence is therefore unnecessary.

Every matcher change is evaluated against a versioned historical corpus containing positives, hard negatives, overlapping services, generic AWS prose, punctuation variants, Unicode, HTML, edits, and security guidance. Promotion records precision and recall per service and risk type plus the approved thresholds.

## Environment and route mapping

For each matched service:

- Select enabled environment policies whose profile contains the service.
- Resolve each environment through the exact inventory release.
- Group environment IDs by route ID.
- Sort IDs deterministically.
- Create one candidate for each route.

Customer labels, account IDs, and Regions are display context loaded by the worker. They do not affect candidate identity. Disabled or unconfigured environments never appear.

## AlertCandidate version 3

The schema is authoritative. The contract contains:

- Candidate ID and creation time.
- Audience fingerprint derived from the sorted environment IDs.
- Public announcement source type, canonical IDs, title, summary, URL, observed time, optional publication time, update flag, and sorted provenance.
- Service ID and display name.
- Risk rule ID, risk type, and priority.
- One route ID and sorted environment IDs.
- `relevance_basis = inferred_profile_match`.
- Matched profiles, aliases, terms, fields, human-readable reason, and recommended action.
- Exact release ID, object keys, object version IDs, hashes, schema versions through the manifest, and application version.

Candidate identity is:

```text
SHA256(
  "candidate:v3\0" + revision_id + "\0" + service_id + "\0" + risk_type + "\0" +
  route_id + "\0" + audience_fingerprint
)
```

`audience_fingerprint` is the null-framed SHA-256 digest of `candidate-audience:v1` followed by the sorted environment IDs. Rule ID is excluded so a rule rename does not resend unchanged logical work. A changed environment set creates a new candidate. The candidate byte limit is checked after canonical JSON serialization.

Slack grouping occurs within one candidate: the message groups all matching environments for its route. Separate candidates retain separate delivery records and Slack outcomes.

## DeliveryRequest version 3

A delivery request contains the exact candidate, destination key, creation time, and:

```text
request_id = SHA256("delivery-request:v2\0" + candidate_id)
```

Embedding the candidate prevents the dispatcher from constructing a different payload than the one persisted. The worker still loads the exact release to verify route and display data. The request byte limit remains below the SQS message limit with room for service encoding.

## Durable emission

For each candidate, conditionally write the candidate and its `pending_queue` delivery record. The feed response checkpoint cannot advance until all deterministic candidate IDs from that response are present with valid outbox records. A crash before checkpoint advancement causes safe reconstruction.

If a candidate ID already exists, load and validate the stored candidate rather than replacing it with fields from the current release or rule evaluation. Its original evidence, release, and creation time remain immutable. Reuse its exact stored request when repairing a missing delivery record. An existing item whose identity fields do not match its key is a correctness failure and blocks checkpoint advancement.

Announcement provenance can grow after a candidate was emitted. Historical candidates remain immutable. The announcement record shows the merged current provenance.

## Slack rendering

The Slack message includes:

- A top-level plain fallback string.
- Priority and potentially relevant wording.
- Announcement title and source link.
- Service and risk type.
- Why the rule matched.
- A bounded environment/customer summary with omitted-count text.
- Recommended review action.
- Publication time when present, otherwise observation time.
- Candidate ID in a compact context field for support.

Render untrusted content in `plain_text` blocks where available. Escape Slack control characters in fallback and link text. Source URLs must pass the same canonical HTTPS policy before becoming links. High-priority mentions use only configured user-group IDs.

If a message exceeds configured limits, apply deterministic field truncation. If it still exceeds the contract, mark delivery `failed_terminal`; do not silently omit identity, evidence, or source URL.

## Slack outcomes

- Documented success becomes `posted`.
- `429` with a valid bounded `Retry-After` becomes `failed_retryable` with a future next action.
- Reviewed transient server responses become `failed_retryable` with bounded exponential backoff and jitter derived without changing identity.
- Authentication, revoked hook, invalid channel, or permanent payload errors become `failed_terminal`.
- Timeout, connection loss after bytes may have been sent, worker death during a network attempt, or expired sending lease becomes `delivery_unknown`.

Unknown outcomes receive no automatic retry. An operator may inspect Slack and record a manual replay decision.

## Incoming-webhook controls

The secret value is parsed at runtime. Require HTTPS, port 443, an approved host, the expected Slack service path, no fragment or user info, and no redirects. Resolve and connect using the same anti-rebinding controls as feed acquisition. Never log the URL.

## Bot-token controls

The route uses a configured channel ID and deployment-wide workspace ID. `destination_key` is the normalized workspace/channel pair. The worker calls only the required Slack posting method and stores a returned message timestamp when present.
