Explainable AWS public-change alerting, designed through explicit decisions.

How AI was used See the AWS flow

AWS Public Change Alerting turns approved public announcements into route-scoped Slack candidates for teams operating repeated AWS stacks. Each candidate records the source revision, matching evidence, potentially relevant environments, and exact policy release.

Value

A focused review queue from a broad AWS announcement stream.

General feed readers collect AWS news. This design adds declared service profiles, risk rules, environment mapping, and team routes. Platform teams receive explainable review candidates without granting the service access to customer AWS accounts.

Useful output

Evidence attached to every candidate

The result names the public source, service, risk evidence, announcement revision, route, audience, application version, and immutable configuration release.

Honest boundary

Potential relevance requires human review

Public announcements cannot prove account or resource impact. Operators use the candidate as a review prompt and confirm applicability with their existing AWS tooling.

OpenAI Codex, using GPT-5, worked as an architecture and engineering assistant under Lila Brooks's direction. The work was iterative: define the product question, challenge ambiguous boundaries, record decisions, encode contracts, run checks, and revise until the repository agreed with itself.

AWS architecture

Deterministic matching before a durable Slack delivery boundary.

The design uses Lambda for scheduled and event-driven work, S3 for immutable releases and feed snapshots, DynamoDB for source and delivery state, SQS FIFO for destination-ordered transport, and CloudWatch plus SNS for operations.

01 · Acquire

Fetch approved public sources

The watcher applies HTTPS allowlists, DNS and IP checks, TLS verification, redirect rejection, response limits, parser limits, and per-feed checkpoints.

02 · Match

Create route-scoped evidence

Normalized announcements are matched against service aliases and distinct risk evidence, then mapped through static profiles to sorted route audiences.

03 · Persist

Cross the durable outbox boundary

DynamoDB stores each candidate and delivery request before the watcher advances its feed checkpoint. Replays reconstruct the same logical identity.

04 · Deliver

Order, pace, and recover Slack work

SQS FIFO groups work by destination. The worker applies pacing and records posted, retryable, terminal, or unknown outcomes for recovery.

Why this architecture

The design follows the evidence and failure model.

Thirteen accepted ADRs govern the current architecture. ADR-017 narrowed an earlier design to public-feed intelligence and Slack delivery, removing account-scoped health, cost, and external-adapter paths from the product.

Scope and trust

Public feeds and static profiles

One public evidence class keeps the product narrow. Static mappings provide environment context without cross-account roles, resource discovery, or customer telemetry.

ADR-017 · Product scope
Evidence and identity

Deterministic rules and null-framed hashes

Reviewers can trace why a service and risk matched. Stable identity domains keep replay, edits, route isolation, and dedupe behavior testable.

ADR-002 · Route-scoped identity
State and transport

DynamoDB outbox with SQS FIFO

DynamoDB owns durable intent and outcomes. SQS handles ready-work transport and destination ordering; its time-limited dedupe cannot replace lasting state.

ADR-007 · Durable delivery
Replay and uncertainty

Immutable releases and explicit unknown outcomes

Exact S3 object versions reproduce the policy behind a candidate. Slack timeouts become delivery_unknown because an HTTP call cannot commit atomically with DynamoDB.

Evidence and status

Repository contracts are validated; runtime evidence remains planned.

The current repository proves that its schemas, examples, semantic rules, deterministic identities, documentation, and CI checks agree. It does not yet prove deployed AWS behavior or live Slack delivery.

Available evidence

  • 6 strict schemas with one valid cross-file example bundle
  • Semantic validation, deterministic identity vectors, and mutation tests
  • Repository quality, page drift, link, Mermaid, and HTML checks

Required implementation evidence

  • Python runtime and Terraform bootstrap and service roots
