Reference architecture

Public AWS changes, reduced to explainable review work.

AWS Public Change Alerting turns approved public announcements into route-scoped Slack candidates for teams operating repeated AWS stacks. Every candidate carries the source revision, matching evidence, potentially relevant environments, and exact policy release that produced it.

Value proposition

A smaller, traceable review queue from a broad public stream.

AWS publishes service changes across What's New, blogs, and security bulletins. General feed readers can collect those items, yet they lack the service profiles, risk rules, environment mapping, and team routes needed to make each item actionable. This architecture supplies that context without customer-account permissions.

For platform teams

Review the changes that match declared services

Static profiles connect service definitions to environments. A candidate identifies the environments that are potentially relevant and routes the review to the right Slack destination.

For reviewers

See the reason behind every candidate

Each result names the matched service, risk type, rule evidence, public source, announcement revision, route, audience, application version, and immutable configuration release.

For operators

Keep delivery state recoverable

DynamoDB retains candidates, outbox work, pacing, retries, and terminal outcomes. SQS carries ready work. Slack responses that may have succeeded remain visible as delivery_unknown.

Evidence boundary

Treat relevance as an inference

A public announcement plus a declared service profile establishes potential relevance. Operators confirm account or resource impact with their existing account-specific tools.

Product boundary: approved AWS RSS and Atom feeds, deterministic matching, static profile and route mapping, durable candidate history, and Slack delivery.

Customer-account access, account telemetry, spend collection, security-finding ingestion, remediation, ticketing, and incident ownership remain outside this repository.

Processing flow

Deterministic processing with a durable delivery boundary.

The watcher turns public source material into immutable route-scoped candidates. Delivery proceeds through a durable outbox, destination-ordered transport, and a worker that records every known or ambiguous Slack outcome.

  1. 01

    Publish one immutable release

    Reviewed configuration and inventory are validated together, written to exact S3 object versions, hashed, and promoted through a compare-and-swap manifest.

  2. 02

    Acquire public feeds safely

    The watcher fetches approved HTTPS RSS and Atom sources with host, DNS, TLS, redirect, response-size, parser, and timeout controls.

  3. 03

    Normalize and match evidence

    Items are normalized, coalesced by canonical URL, tracked by content revision, and matched using service aliases plus distinct positive risk evidence.

  4. 04

    Build route-scoped candidates

    Profiles map a matched service to enabled environments. Environments are grouped by route, sorted, and bound into deterministic candidate identities.

  5. 05

    Cross the durable outbox boundary

    The candidate and delivery request become durable before feed validators advance. A dispatcher sends due work to an SQS FIFO group for the Slack destination.

  6. 06

    Deliver and retain the outcome

    The Slack worker verifies the exact release, applies pacing, posts once for its claim, and records success, retryable failure, terminal failure, or an unknown outcome.

Architecture decisions

The design follows explicit evidence, state, and trust boundaries.

The current architecture is governed by 13 accepted decision records. ADR-017 narrowed an earlier multi-lane design to public-feed intelligence and Slack delivery, archiving the AWS Health, cost, and external-adapter paths. The resulting product answers one question well: which public AWS changes deserve review for the services a team has declared?

Scope

Public feeds form the evidence source

Public announcements have one consistent evidence model and require no customer-account trust. The architecture labels environment matches as potentially relevant and leaves confirmed impact to account-specific tooling.

ADR-017 · Public-feed-only scope
Explainability

Rules are deterministic and reviewable

Service aliases and risk terms use defined normalization, phrase boundaries, positive and excluded evidence, and distinct match spans. Replays produce the same identity from the same normalized inputs.

ADR-009 · Acquisition and matching
Isolation

Candidate identity stops at the route boundary

One candidate contains one route and its complete sorted environment audience. This keeps customer and team context from crossing Slack destinations during delivery or replay.

ADR-002 · Route-scoped identity
State ownership

DynamoDB owns delivery state

The outbox records intent and outcome before SQS transports ready work. Time-limited queue dedupe cannot replace durable candidate, dispatch, retry, pacing, and recovery state.

ADR-007 · Durable outbox and worker
Delivery semantics

Ambiguity becomes an explicit state

A Slack HTTP call cannot commit atomically with DynamoDB. A timeout or expired sending lease becomes delivery_unknown and waits for operator review instead of risking an automatic duplicate.

ADR-004 · Explicit Slack guarantees
Reproducibility

Every result names its exact release

Configuration and inventory are published under immutable object versions. Candidates retain their hashes, schema versions, release ID, and application version for later replay or investigation.

ADR-014 · Immutable releases

System properties

The architecture makes its guarantees and limits testable.

Identity, configuration shape, route isolation, retries, and state transitions are expressed as contracts or acceptance cases. Operational claims require deployment evidence before production use.

Identity

Stable across replay

Null-framed SHA-256 domains identify announcements, revisions, audiences, candidates, requests, and queue dispatches. Material source or audience changes create new work.

Configuration

Immutable and exact

Owned schemas reject unknown fields. Runtime loads exact configuration and inventory object versions and verifies their hashes before processing or delivery.

Security

Credentials stay with delivery

The feed watcher and dispatcher cannot read Slack secrets. The Slack worker reads only configured secret resources and performs the external delivery call.

Operations

Failures remain visible

Feed freshness, outbox age, queue age, terminal failures, unknown outcomes, DLQ depth, and recovery work have explicit metrics and alarm requirements.
